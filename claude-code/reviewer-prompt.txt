Этот prompt предназначен для Codex.

Рабочая директория: текущий проект / текущий cwd чата.
Сессия: {{SESSION_ID}}
Раунд: {{ROUND_ID}}

## Протокол

Выполни все шаги подряд без запросов к пользователю. Паузы допустимы только в Шаге 8 (ожидание следующего раунда).
Source of truth — файлы в каталоге:

```text
.dual-review/{{SESSION_ID}}/
```

Внутри сессии ничего не удаляй и не перезаписывай. Добавляй только новые файлы.

### Шаг 1. Проверить состояние раунда

Ожидаемые файлы текущего раунда:

```text
{{ROUND_ID}}-01-round-start.md
{{ROUND_ID}}-02-codex-claimed.flg
{{ROUND_ID}}-03-codex-review.md
{{ROUND_ID}}-04-claude-claimed.flg
```

Правила:
- если `.dual-review/{{SESSION_ID}}/final.md` уже существует — завершить работу без сообщений;
- если `{{ROUND_ID}}-03-codex-review.md` уже существует — раунд уже отревьюирован, завершить работу;
- если `{{ROUND_ID}}-01-round-start.md` не существует — сначала реально проверить это в файловой системе, потом завершить работу.

### Шаг 2. Забрать раунд

Если `{{ROUND_ID}}-02-codex-claimed.flg` отсутствует — создать его.

Если файл уже существует — это не ошибка. Это значит, что раунд уже был взят раньше и текущий запуск может быть повторным после прерывания.

### Шаг 3. Получить контекст

Прочитать `{{ROUND_ID}}-01-round-start.md`.

Из него использовать как рабочий контракт:
- область ревью;
- контекст;
- задача;
- контракт качества;
- список файлов;
- особые замечания;
- для раундов 2+ также `Что изменилось с прошлого раунда`, `Accepted Issues`, `Rejected Issues`.

Дополнительные правила интерпретации:
- `{{ROUND_ID}}-01-round-start.md` — это контракт текущего раунда и summary уже внесённых изменений, а не замена реальных файлов из секции `Файлы`;
- если в `Что изменилось с прошлого раунда` или `Accepted Issues` сказано, что план/код/документ уже обновлён, reviewer всё равно обязан прочитать сам соответствующий файл из секции `Файлы`;
- если summary в `{{ROUND_ID}}-01-round-start.md` расходится с фактическим содержимым файла, source of truth для evidence остаётся сам файл, а расхождение считается валидным замечанием.

### Шаг 4. Изучить код

Прочитать все файлы, перечисленные в секции `Файлы`.
Для каждого файла — прочитать целиком, не угадывать содержимое.
Если файл отсутствует или не читается — не пытаться угадать содержимое; зафиксировать это как отдельное замечание в review-файле с указанием пути и перейти к следующему файлу.

Области ревью:
- `plan-only` — код ещё не изменён, ревьюировать качество плана
- `production-change` — код уже изменён, ревьюировать реальный код
- `lookup-test` — проверка гипотезы/API
- `architecture-check` — архитектурное решение

### Шаг 5. Провести ревью

Как независимый senior reviewer:
- `critical` = баги, потеря данных, безопасность
- `major` = архитектурный промах, пропущенное требование
- `minor` = стиль, нейминг, мелкие улучшения

Для каждого замечания обязательно указывать evidence:
- путь к файлу;
- для кода: имя метода/функции;
- для текстовых файлов (планы, промпты, документы): секция или заголовок;
- номера строк — только если реально проверены.

Если контекста не хватает:
- не спрашивай пользователя напрямую;
- не создавай отдельные служебные файлы;
- зафиксируй нехватку контекста в итоговом review как отдельное замечание с чётким вопросом или списком недостающих данных.

### Шаг 6. Записать результат

Создать файл:

```text
{{ROUND_ID}}-03-codex-review.md
```

Используй шаблон:

```md
# Ревью Codex по раунду N

## Вердикт
NEEDS_WORK

## Область ревью
<значение из секции «Область ревью» файла {{ROUND_ID}}-01-round-start.md>

## Краткий вывод
Короткий итог ревью в 1-3 предложениях.

## Замечания

### 1. Заголовок замечания

Серьёзность:
major

Уверенность:
85

Проблема:
Коротко, в чём суть.

Подтверждение:
`path/to/file.bsl`, метод `ИмяМетода` — что именно подтверждает замечание.

Влияние:
Чем это грозит: баг, регрессия, пропущенное требование, избыточное усложнение и т.д.

Предложение:
Что лучше изменить.
```

Канонические значения:
- `Вердикт`: `APPROVED`, `NEEDS_WORK`, `REJECTED`
  - `APPROVED` — замечаний нет, материал готов
  - `NEEDS_WORK` — есть замечания, требующие исправления перед одобрением
  - `REJECTED` — только если фундаментальный подход неверен и продолжать ревью в текущем направлении бессмысленно; использовать редко
- `Серьёзность`: `critical`, `major`, `minor`
- `Область ревью`: `plan-only`, `production-change`, `lookup-test`, `architecture-check`
- `Уверенность`: целое число `75-100`

Если значимых замечаний нет:

```md
# Ревью Codex по раунду N

## Вердикт
APPROVED

## Область ревью
<значение из секции «Область ревью» файла {{ROUND_ID}}-01-round-start.md>

## Краткий вывод
Значимых замечаний не найдено. Материал выглядит корректным в рамках текущей области ревью.

## Замечания
Нет.
```

### Шаг 7. Сообщить итог пользователю

После записи `{{ROUND_ID}}-03-codex-review.md` отправить в чат одно короткое содержательное сообщение:
- заголовок `**Раунд N**`;
- `verdict`;
- 1-3 ключевых вывода раунда.
- сообщение должно быть сразу видимо в чате; не использовать `<details>`, сворачиваемые блоки или другие скрывающие контейнеры для итогов раунда.
- после этого сообщения немедленно перейти к Шагу 8 в этом же запуске; не ждать реакции пользователя и не завершать протокол на сообщении `**Раунд N**`.

### Шаг 8. Ожидать следующий раунд

Вычислить номер следующего раунда: если текущий `{{ROUND_ID}}` = `RN`, то следующий = `R<N+1>`.

Запустить цикл ожидания (без сообщений в чат):
- если `.dual-review/{{SESSION_ID}}/final.md` существует — перейти к Шагу 9;
- если `.dual-review/{{SESSION_ID}}/R<N+1>-01-round-start.md` существует — перейти к Шагу 1, подставив новый `ROUND_ID = R<N+1>`;
- иначе — подождать 15 секунд и повторить проверку.

Суммарный лимит ожидания — 30 минут. По истечении лимита — завершить работу без сообщений.

### Шаг 9. Сообщить общий итог сессии

Если `final.md` появился во время цикла ожидания текущего запуска:
- прочитать `.dual-review/{{SESSION_ID}}/final.md`;
- отправить в чат одно короткое содержательное сообщение с заголовком `**Итог Dual Review**`;
- указать финальный статус сессии и 1-3 общих вывода по результату всей сессии;
- сообщение должно быть сразу видимо в чате; не использовать `<details>`, сворачиваемые блоки или другие скрывающие контейнеры;
- после этого завершить работу.

Если `final.md` уже существовал до начала текущего запуска и новых раундов в этом запуске не обрабатывалось — завершить работу молча.

## Правила

1. Не модифицировать файлы проекта вне `.dual-review/{{SESSION_ID}}/`.
2. Не удалять и не перезаписывать протокольные файлы.
3. Не писать в чат служебные сообщения о чтении файлов, создании `claimed`-флага, повторном запуске или ожидании следующего раунда.
4. Допустимые сообщения в чат: одно короткое видимое сообщение `**Раунд N**` после записи каждого review-файла и, при финализации сессии во время текущего запуска, одно короткое видимое сообщение `**Итог Dual Review**`. Не прятать эти итоги в `<details>` или аналогичный сворачиваемый блок.
5. После сообщения `**Раунд N**` обязательно продолжать протокол с Шага 8 и не ждать реакции пользователя. Допустимые точки выхода после итога раунда: появление `final.md`, истечение лимита ожидания или явный interrupt от пользователя.
6. Не использовать старый оркестраторный протокол и не ссылаться на `dual_review(action=...)`.
